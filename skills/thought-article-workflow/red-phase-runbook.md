# RED Phase Runbook

这份文件用来指导后面正式测试 `thought-article-workflow`。

目标不是证明它“看起来合理”，而是先找出它最容易失败的地方。

## 为什么先做这个

因为这个 skill 最大的风险不是“不会写”，而是：

- 太快进入改写
- 帮你润色了，但没帮你想清楚
- 学标杆文章时学歪了
- 平台适配时把核心观点写散了

所以在写正式 `SKILL.md` 之前，先盯住这些失败模式。

## 测试对象

后面测试时，优先用这三类真实材料：

### 1. Get 清稿

最适合作为主测试材料，因为它最接近日常使用场景。

### 2. AI 初稿

看 skill 能不能识别“像文章，但核心没立住”的问题。

### 3. 自己已经比较满意的一篇

看 skill 会不会乱改，把已经成立的地方改坏。

## 核心失败模式

### 失败模式 1：太早改写

表现：

- 上来直接润色或重写
- 没先说清文章真正想讲什么

### 失败模式 2：把感受误当洞察

表现：

- 语言变漂亮了
- 但真正的判断仍然模糊

### 失败模式 3：照着喜欢的文章写偏

表现：

- 口气像别人
- 但不再像用户自己

### 失败模式 4：没标出缺支撑的地方

表现：

- 流畅
- 但没有指出哪些地方该补数据、案例或原文

### 失败模式 5：双平台输出没有母稿

表现：

- 直接给公众号版和小红书版
- 核心论点前后不一致

## 每次测试都要看什么

### 第一反应

它做的第一件事是什么？

如果第一步不是诊断，就要高度警惕。

### 诊断质量

它能不能说清：

- 这篇文章真正的核心命题
- 哪些地方没想透
- 哪些地方缺材料
- 哪些地方只是表达问题

### 风格保护

它有没有把你改成：

- 老师
- 知识博主
- 空泛励志型账号

### 平台处理

它有没有先母稿再适配，而不是两边乱飞。

## 通过标准

一个合格的正式 skill，至少应该稳定做到：

- 先诊断，再改写
- 先给问题地图，再给结构提纲
- 明确指出缺想法和缺材料的地方
- 学标杆文章时，能区分“该学什么”和“别学什么”
- 双平台时，先出母稿
- 反复出现的偏好会沉淀，不会每次从零开始

这份 runbook 已经可以直接作为后续正式测试的依据。

等真实跑过几轮后，再根据实际失败情况反向收束进最终 `SKILL.md`。
